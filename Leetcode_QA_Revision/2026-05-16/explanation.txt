Title: Find Minimum in Rotated Sorted Array II
URL: https://leetcode.com/problems/find-minimum-in-rotated-sorted-array-ii/

LLM Explanation:
### Method: Binary Search with Duplicate Handling

This problem is a variation of "Find Minimum in Rotated Sorted Array," with the added complexity of duplicate elements. The core approach is still a modified binary search.

1.  **Binary Search Setup**: We initialize two pointers, `left` and `right`, at the beginning and end of the array, respectively. The search continues as long as `left < right`.

2.  **Comparison Logic**: In each iteration, we calculate a `mid` index and compare `nums[mid]` with `nums[right]` to decide which half of the array to discard.
    *   **`nums[mid] < nums[right]`**: This indicates that the subarray from `mid` to `right` is sorted in non-decreasing order. The minimum element cannot be in the `(mid, right]` range. It could be `nums[mid]` or an element to its left. Thus, we can safely narrow our search space to the left half by setting `right = mid`.
    *   **`nums[mid] > nums[right]`**: This implies that the pivot point (the smallest element) lies in the right subarray `(mid, right]`. The subarray from `left` to `mid` is sorted but contains elements larger than `nums[right]`. Therefore, we can discard the left half and `mid` by setting `left = mid + 1`.

3.  **Handling Duplicates (The Ambiguous Case)**:
    *   **`nums[mid] == nums[right]`**: This is the crucial case introduced by duplicates. We cannot determine whether the minimum is to the left or right of `mid`. For instance, in `[3, 3, 1, 3]`, the minimum is to the right of `mid`, while in `[1, 3, 3, 3]`, the minimum is to the left.
    *   **Resolution**: Since we are uncertain, we cannot discard half of the array. However, we can safely shrink the search space by one element. By decrementing `right` (i.e., `right -= 1`), we discard `nums[right]`. This is a safe move because `nums[mid]` has the same value, so we are not losing a potential minimum candidate. This step ensures progress, although it might be slow.

4.  **Termination**: The loop continues until `left` and `right` converge to the same index. This index will point to the minimum element in the array, which is then returned.

### Complexity Analysis (Follow-up Answer)

*   **Time Complexity**:
    *   **Average Case**: O(log n). In most cases, we discard half of the search space in each step.
    *   **Worst Case**: O(n). The presence of duplicates affects the runtime complexity. In the worst-case scenario, such as an array with many identical elements (e.g., `[1, 1, 1, ..., 1, 0, 1]`), the condition `nums[mid] == nums[right]` might occur repeatedly. Each time, we only reduce the search space by one element (`right -= 1`), causing the algorithm to degrade to a linear scan. This is a change from the O(log n) guarantee in the version without duplicates.

*   **Space Complexity**: O(1), as the algorithm uses only a constant amount of extra space for pointers.

LLM Testcase Validation:
- Testcase 1: `[1,3,5]`
  - `left=0, right=2, mid=1`. `nums[1]=3 < nums[2]=5`. `right` becomes `1`.
  - `left=0, right=1, mid=0`. `nums[0]=1 < nums[1]=3`. `right` becomes `0`.
  - `left` is now `0`, `right` is `0`. The loop `while left < right` terminates.
  - Return `nums[left]` which is `nums[0] = 1`.
  - **Output: 1. Correct.**

- Testcase 2: `[2,2,2,0,1]`
  - `left=0, right=4, mid=2`. `nums[2]=2 > nums[4]=1`. `left` becomes `mid + 1 = 3`.
  - `left=3, right=4, mid=3`. `nums[3]=0 < nums[4]=1`. `right` becomes `mid = 3`.
  - `left` is now `3`, `right` is `3`. The loop `while left < right` terminates.
  - Return `nums[left]` which is `nums[3] = 0`.
  - **Output: 0. Correct.**
