Title: Count Submatrices with Top-Left Element and Sum Less Than k
URL: https://leetcode.com/problems/count-submatrices-with-top-left-element-and-sum-less-than-k/

LLM Explanation:
The user wants to solve the "Count Submatrices with Top-Left Element and Sum Less Than k" problem.

### Method Explanation

The problem asks for the number of submatrices that start at the top-left corner `(0,0)` and have a sum less than or equal to `k`. Any such submatrix is uniquely defined by its bottom-right corner `(r, c)`. The task is therefore equivalent to finding the number of pairs `(r, c)` for which the sum of the submatrix from `(0,0)` to `(r,c)` is at most `k`.

A naive approach of recalculating the sum for each possible bottom-right corner would be too slow, leading to a complexity of O(m² * n²). A much more efficient method is to use a 2D prefix sum array.

1.  **Core Idea: 2D Prefix Sum**
    We can compute a prefix sum matrix where each cell `(r, c)` stores the sum of all elements in the rectangle from `(0,0)` to `(r,c)`. This allows us to find the sum of any required submatrix in O(1) time after an initial O(m*n) preprocessing step.

2.  **In-place Calculation**
    To optimize for space, instead of creating a separate prefix sum matrix, we can modify the input `grid` in-place. As we iterate through the grid row by row, and column by column, we update `grid[r][c]` to become the prefix sum of the submatrix ending at `(r,c)`. The calculation relies on the standard prefix sum recurrence:
    `prefix_sum(r,c) = grid(r,c) + prefix_sum(r-1,c) + prefix_sum(r,c-1) - prefix_sum(r-1,c-1)`
    When we compute the value for `grid[r][c]`, the values `grid[r-1][c]`, `grid[r][c-1]`, and `grid[r-1][c-1]` have already been converted to their respective prefix sums, so the formula works correctly in-place.

3.  **Algorithm and Optimization**
    The algorithm iterates through each cell `(r, c)` of the grid:
    - It first calculates the prefix sum for `grid[r][c]`.
    - It then checks if this sum is less than or equal to `k`. If it is, a counter is incremented.
    - An important optimization is possible because the problem constraints state that all grid elements are non-negative. This implies that for any fixed row `r`, the prefix sums `prefix_sum(r, c)` will be a non-decreasing sequence as `c` increases.
    - Consequently, if we find that `prefix_sum(r, c) > k`, we can be certain that `prefix_sum(r, c+1)`, `prefix_sum(r, c+2)`, etc., will also be greater than `k`. We can therefore `break` the inner loop and proceed to the next row, avoiding unnecessary computations.

### Complexity Analysis
-   **Time Complexity:** O(m * n), where `m` is the number of rows and `n` is the number of columns. We visit each cell of the grid exactly once.
-   **Space Complexity:** O(1) extra space, as the prefix sums are calculated by modifying the input grid in-place.
