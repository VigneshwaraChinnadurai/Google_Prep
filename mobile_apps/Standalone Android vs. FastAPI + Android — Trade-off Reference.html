Standalone Android vs. FastAPI + Android — Trade-off Reference

Decision record for the Job Finder project. Captures why the standalone build exists, what it costs, and when to pick which. Written for future-me who's forgotten the reasoning.

Last updated: 2026-05-04 Author: Vigneshwara Chinnadurai Projects this applies to:

  • usecase_2/agentic_job_finder_mobile — FastAPI backend + Android client
  • usecase_3/job_finder_standalone — On-device Android app

TL;DR

FastAPI version Standalone
Best for Daily job hunting from your laptop Demo, distribution, offline-first reference
Setup time ~10 min (Python env + Gradle sync) ~5 min (Gradle sync only)
Companies covered ~78 (Greenhouse + Lever + Workday + generic/Playwright) 18 (Greenhouse + Lever only)
LLM cost Server-paid User-paid (their own API key)
Devices Multi-device Single device
iOS path Trivial (swap client) Hard (KMP rewrite)

Default recommendation: use the FastAPI version. Pick standalone only when you've explicitly hit a constraint that makes the backend version untenable.

Why this document exists

"Make it standalone" sounds like a small architecture change. It isn't. Going standalone forces you to give up specific capabilities that aren't obvious until you look at them in detail. This doc lists them so you don't re-discover them by surprise.

The blockers, ranked

🔴 Hard blockers — these don't have workarounds

1. Playwright cannot run on Android

Playwright needs Chromium + Node.js + a real OS. There is no Android port and no plausible path to one. Any company in companies.yaml with ats: generic becomes unreachable on the standalone build.

Concrete cost: ~25 of the 78 Bangalore companies disappear. Lost names include Google, Apple, Meta, Microsoft, Flipkart, Swiggy, Zomato, Zoho, Razorpay, Dream11, Mu Sigma, InMobi, BigBasket, Paytm, PhonePe, and several others that use bespoke career portals.

Workaround paths (all imperfect):

  • Write per-portal HTTP extractors in Kotlin → high effort per company, fragile against redesigns. Worth it only for the top 3-5 names.
  • Run Playwright on a tiny VPS and call it from the app → defeats the point of "standalone".
  • Just accept the smaller list → what the standalone build does today.

2. Workday tenants reject mobile UA / residential IPs

Workday's public job-feed endpoints accept POST requests, but most production tenants (especially enterprise ones) sit behind WAFs that flag mobile User-Agent strings on residential IPs. You'll see a mix of 200s, 403s, and silent zeros.

Concrete cost: Workday-hosted companies (Salesforce, SAP, ServiceNow, Workday itself, Walmart, Target, Visa, Mastercard, Adobe, Oracle, IBM, Cisco, Intel, NVIDIA, Qualcomm, HP, HPE, Dell, VMware) all become unreliable.

Workaround: Drop Workday from the standalone build entirely. Better than shipping silently broken results.

3. LLM API key has nowhere safe to live

Three options exist, all bad:

Option What you get What you lose
User pastes their own key Honest about who pays Can't ship to non-technical users
Bundle a key in the APK Works for everyone Key is extractable in 5 minutes; one viral install = bankruptcy
Proxy through your server Centralized control Defeats the entire point of "standalone"

The standalone build picks "user pastes their own key", stored in EncryptedSharedPreferences with MasterKey.AES256_GCM. This is safe against casual file-system browsing and untrusted apps on the same device. It is not safe against a determined attacker with root access.

4. PDF / DOCX libraries don't exist for Python on Android

pypdf and python-docx are Python-only. Android needs JVM equivalents.

The chosen substitutes:

  • com.tom-roush:pdfbox-android:2.0.27.0 — Tom Roush's Android port of Apache PDFBox. Works. Adds ~5 MB.
  • org.apache.poi:poi-ooxml:5.2.5 — Apache POI for .docx. Works. Adds ~10 MB.

Concrete cost: APK size goes from ~6 MB to ~18 MB.

🟠 Soft blockers — solvable but with real UX cost

5. Battery and data cost shifts to the user

A full Run on the FastAPI version is "open phone, see results" because the laptop did 200 LLM calls in the background. On standalone, those 200 calls happen on the phone, on the user's data plan, at the cost of their battery.

Empirical numbers (Pixel 7, Wi-Fi, 18 companies, ~100 jobs):

  • Scrape: ~30s, ~2% battery, ~3 MB data
  • Match: ~3 min, ~5% battery, ~500 KB data (LLM responses are small)
  • Total: ~5% battery for a full Run. Acceptable, but not free.

Mitigation: Bound concurrency to 4 in flight. Don't auto-run on app open. Surface "Run" as an explicit user action.

6. Backgrounding requires WorkManager + battery-optimization waivers

Android increasingly resists long-running background work. To do daily overnight refreshes you'd need:

  • WorkManager setRequiresCharging(true)
  • A user-toggle in Settings (opt-in, not default)
  • A foreground notification on some OEMs (Xiaomi, Vivo, Oppo are aggressive)

Decision: v1 doesn't include background refresh. User taps Run when they want results. Adding it later is one file change.

7. companies.yaml is shipped in the APK

Editing the company list = release a new APK = users have to update.

Mitigation paths:

  • Today: Bundle a thoughtful list, accept the rebuild cost.
  • Future: Pull companies.yaml from a public Gist URL on app start, fall back to the bundle on offline / 404. ~30 LOC change.

8. Debugging loop is slower

FastAPI Standalone
Change code Save file Save file
Run change uvicorn --reload (instant) Gradle build + install (15-60s)
Inspect httpx from REPL logcat
Backend logs structured to console scattered across Log.w

This adds friction every iteration. Not catastrophic, but real.

9. iOS path closes

The FastAPI version is iOS-ready by definition: same backend, swap the client to SwiftUI. The standalone version puts scrapers, parsers, and matchers in Kotlin — none of which run on iOS without KMP or rewrite.

Implication: If multi-platform matters, choose FastAPI. Don't convince yourself you'll "port it later" — you won't.

🟡 Inconveniences — annoying but cheap to live with

10. No multi-device

One phone, one cache. If you replace the device, you lose your match history, approval/skip status, and the trained "preferences" implicit in the cached matches.

Mitigation: Room → Backup → Google Drive sync is a real path but adds plumbing. Not in v1.

11. Logs aren't centralized

When the FastAPI version had a flaky scraper, you'd just tail -f uvicorn.log. On standalone, you need:

  • USB debugging on
  • adb logcat | grep com.vc.jobfinder

Or accept that production crashes are diagnosed by user reports, not by you inspecting logs.

12. No structured progress events

The FastAPI version streamed WS /events so the UI could show "Scraping Anthropic — 3 jobs found". The standalone version reproduces this with Flow<ScrapeEvent>, which works the same way for the user but is materially harder to inspect during development (no Postman, no curl-able stream).

What stays the same

It's important to not undersell what survived the transition:

  • The 4-tab UI (Resume / Run / Matches / Contests)
  • All 4 contest platforms (HackerEarth, HackerRank, Unstop, Kaggle) — their endpoints work fine from mobile because they're public-facing by design
  • Tracked-company matching against companies.yaml ("Google LLC" → "Google" via the same suffix-stripping regex)
  • Cover-letter generation per match
  • The Approve / Skip / Apply workflow
  • "Never auto-submit" rule

The data flow is identical; only the location of execution moved.

Decision matrix — when to pick which

Your situation Pick
You're job-hunting from your laptop, daily FastAPI
You want broad ATS coverage including Workday FastAPI
You need Google / Apple / Meta / Flipkart / Swiggy specifically FastAPI
You want one shared cache across phone + tablet FastAPI
You're flying / commuting / want to triage in the wild Standalone
You want to demo the project to someone without setting up infra Standalone
You want to share the APK with a friend who doesn't run Python Standalone
You want a portfolio reference for "offline-first agentic Android" Standalone
You eventually want iOS FastAPI
You want to learn KMP later Standalone (Kotlin codebase is the starting point)

Cost summary in dollars and minutes

Assuming 18 companies, ~100 jobs scraped, Gemini 2.5 Flash, India pricing:

FastAPI version Standalone
Initial setup 10 min 5 min
Per Run, time 5 min 5 min
Per Run, battery ~0% (laptop) ~5% (phone)
Per Run, LLM cost $0.02 (you pay) $0.02 (user pays)
Daily runs × 30 days $0.60 / month $0.60 / month
Hosting $0 (laptop) $0
Apple Developer (if iOS) $99 / year Not applicable (no iOS path)

LLM cost is identical; the only delta is who pays. For personal use, this is irrelevant — you're paying yourself either way. For distribution, it's the entire question.

What the standalone trade-off taught me

Three lessons worth remembering:

  1. "Just put the backend on the phone" is never just that. Every server-side capability has hidden mobile constraints — battery, IP reputation, native library availability, key storage. The cleanest architecture is usually: small client, smart server, even when "the server" is a $5 VPS.

  2. The right answer to "make it standalone" is often "what specifically forced you to ask?" If the answer is "I want to demo it" → ship a standalone build. If it's "the laptop is annoying" → fix the laptop UX with auto-start scripts. Going standalone for cosmetic reasons buys real costs for fake benefits.

  3. Don't pretend the standalone version is feature-equivalent. Every README, every comparison should explicitly call out what's missing. The temptation is to say "v2 will have Workday support" — and v2 might never come.

Forward-looking notes

If I revisit this project in 2027:

  • Check whether Playwright-on-Android became viable (it likely won't).
  • Re-evaluate Workday's mobile-UA blocking — they may relax it.
  • Consider whether KMP has matured enough to share scraper logic across iOS + Android + JVM. As of 2026, it's close but still a meaningful commitment.
  • Reconsider whether Anthropic / OpenAI / Google have launched legitimate on-device LLM endpoints with free or fixed-cost tiers — this would change the API key calculus completely.

If any of those flip, the standalone version becomes more attractive than it is today.
